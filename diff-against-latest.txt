+17. COPY() NOW PRESERVES COMPILATION STATE
+------------------------------------------
+File: javatools/src/main/java/org/xvm/compiler/ast/AstNode.java
+
+Problem: The copy() method using withChildren() created fresh nodes via
+constructors, losing parent, stage, and component references. This broke
+test compilation in PropertyDeclarationStatement which creates a copy to
+test-compile an initializer.
+
+Original clone() behavior: super.clone() shallowly copied ALL fields
+(parent, stage, component), then deep-copied children via getChildFields().
+
+New copy() fix (lines 283-286):
+  // preserve the parent and stage from the original - this allows the copy to be
+  // used in the same compilation context (e.g., for test compilation)
+  copy.m_parent = this.m_parent;
+  copy.m_stage  = this.m_stage;
+
+File: javatools/src/main/java/org/xvm/compiler/ast/ComponentStatement.java
+
+Added copy() override to also preserve the component field (lines 19-26):
+  @Override
+  public ComponentStatement copy() {
+      ComponentStatement copy = (ComponentStatement) super.copy();
+      copy.component = this.component;
+      return copy;
+  }
+
+Result: copy() now has the same semantics as clone() without using clone().
+No setters needed after copy - the copy is immediately functional.
+

+Design Principles Applied
+================================================================================
+
+1. IMMUTABILITY AT BOUNDARIES
+   - Constructor parameters use @NotNull annotations
+   - Null replaced with empty collections (List.of())
+   - List.copyOf() ensures immutability of stored data

+2. BACKWARD COMPATIBILITY
+   - Varargs overloads accept arrays, convert to Lists internally
+   - Deprecated getValueArray() returns array for legacy callers
+   - Gradual migration path - no breaking changes

+3. NO SETTERS AFTER COPY
+   - copy() returns fully-functional nodes
+   - Parent, stage, component all preserved
+   - Eliminates mutation after construction

+   - m_listVal could be made final with builder pattern
+   - Lazy initialization should use Lazy.java mechanisms
