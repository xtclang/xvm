# reflect.x and generics.x both parse cleanly now -- both removed from
# this list. The two fixes that closed them out:
#
# 1. C5 LHS via split shift operators: `<<` / `>>` / `>>>` are emitted
#    as `seq('<', token.immediate('<'))` etc. so the lexer never
#    produces a multi-char shift token. This unblocks the LHS of
#    `Function<<Int, String>, <Int>> f = ...`. Closing side already
#    handled by the TYPE_GT external scanner (commit d384f2394). Cost:
#    a 1-element self-conflict on `binary_expression`.
#
# 2. Template interpolation expressions can span multiple physical
#    lines via the parent multiline template's `|` continuation. The
#    scanner now also handles trailing whitespace before the closing
#    `}` of an interpolation -- forms like `$"{y } abc"` work -- and
#    emits a MULTILINE_CONTINUATION extras token covering `\n + hspace
#    + |` when persistent state shows we're inside a multiline-template
#    interpolation expression (depth counter bumped on
#    MULTILINE_EXPR_START, popped on TEMPLATE_EXPR_END). Without this
#    the `|` got lexed as bitwise-OR, breaking method chains like
#    `{method.as(Type).\n   |   invoke()}`.
#

# === callTests.x — bare-ident + typed-decl in tuple_assignment ===
#
# Failing shape: `(i1, Int i2) = testMultiReturns(0);` -- a tuple
# assignment whose first element is a bare existing-variable
# identifier (`i1` was declared earlier) and whose second is a fresh
# typed declaration (`Int i2`). The grammar's
# `tuple_assignment_element` alternation accepts both the typed form
# (`Type identifier`) and bare expressions (`$._expression`), so both
# elements *should* match.
#
# However, GLR consistently commits to the alternative interpretation:
# `(i1, Int i2)` parses as a `tuple_type` (with `i1` read as a
# type_name and `Int i2` as a typed tuple_type_element), making the
# whole line a `variable_declaration` with `(i1, Int i2)` as the type
# and `testMultiReturns` as the variable name -- which then ERRORs
# on `=`.
#
# Tried (none worked):
#   - `prec(2)` on `tuple_assignment` itself
#   - `prec.dynamic(1)` on the `$._expression` alt of `tuple_assignment_element`
#   - 4-way conflict adding `_for_tuple_destructure_element`
#   - `prec.dynamic(10)` on the entire `tuple_assignment_element` rule
#   - `prec.dynamic(10, prec(2, ...))` on the entire `tuple_assignment` rule
#     (parse time tripled, confirming GLR explored more paths, but
#     the result was unchanged -- prec.dynamic is not consulted at
#     the deterministic-choice point)
#   - `prec.dynamic(-1)` on `tuple_type_element` to disprefer it
#     when GLR has tuple_assignment_element alive in parallel:
#     parse time went up (more paths explored) but the deterministic
#     LALR commit to `variable_declaration` was unchanged.
#   - explicit `[$.tuple_assignment, $.variable_declaration]` conflict
#     -> tree-sitter reports "unnecessary conflict"
#   - explicit `[$.tuple_assignment, $.tuple_type]` and
#     `[$.tuple_assignment, $.parenthesized_type]` conflicts
#     -> both reported as "unnecessary conflicts"
#
# Diagnostic insight from the "unnecessary conflict" warnings:
# tree-sitter's LALR(1) generator never sees a generation-time
# conflict between these container-level rules. It commits to
# `variable_declaration` (with `(...)` as a tuple_type type_expression)
# deterministically when seeing `(` at statement start. The
# element-level conflict declarations and prec.dynamic biases never
# get a chance to apply because the container-level decision was
# already made.
#
# Working cases (for reference -- show what *does* parse correctly):
#   - `(String s, Int i) = expr;` -- both elements explicitly typed,
#     so first-element shape commits the parser to tuple_assignment
#     before any GLR ambiguity.
#   - `(Int? i1N, Int i2N) = expr;` -- first element typed with `?`
#     suffix, same reason.
#   - `(n, c) = name();` -- bare-ident-only, parses as
#     `assignment_expression` (left = `tuple_expression`).
#   - `(Int bucketCount, this.growAt) = ...;` -- typed first, member-
#     access second; first-element shape disambiguates.
#
# The fix likely requires either:
#   (a) a structural rewrite that disambiguates earlier (e.g. lift the
#       `(bare_ident, Type ident, ...) = ...` shape into its own
#       container that the LALR generator can statically distinguish);
#   (b) an external scanner that peeks past matching `)` for `=` and
#       emits a TUPLE_ASSIGN_LPAREN token that uniquely starts
#       tuple_assignment, forcing LR commitment;
#   (c) a workaround in `tuple_assignment_element` that uses an alias
#       with high-prec.dynamic so the bare-ident path doesn't even
#       reduce as identifier-via-type-name.

# === literals.x — one remaining error: bare absolute-path primary expression ===
#
# Most of literals.x's gaps were closed across this session's
# commits (relative_path_literal in typed_literal, bare `./` and
# `../` accepted, multiline hex byte literal `#|...|`, date_literal,
# time_literal, richer duration_literal regex). The single remaining
# error is line 150:
#
#     Path path6 = /allow.as.assert.avoid.public.private.if.for.while.var.val.void.txt;
#
# This is a bare ABSOLUTE path (`/`-prefixed) used as a primary
# expression. There is no rule for it. Adding
# `absolute_path_literal: $ => /\/[\w][\w./-]*/` would conflict with
# spaceless division -- `bytesLen/2` is real production code in
# `lib_ecstasy/.../Source.x` and tree-sitter's longest-match
# tokenization would prefer the path interpretation, breaking the
# division.
#
# The natural fix shape -- an external-scanner-emitted
# ABSOLUTE_PATH_LITERAL gated on `valid_symbols[ABSOLUTE_PATH_LITERAL]`
# (analogous to the TYPE_GT and MULTILINE_CONTINUATION work earlier
# in this branch) -- was implemented and tested end-to-end. The
# scanner correctly emits the token in `typed_literal` context (after
# `Type:`, e.g. line 259 `FileStore fs = FileStore:/archive/moduleTest;`)
# but is *never invoked* at the line-150 position
# `Path path6 = ` ... `/allow.txt`. Instrumenting the scanner with an
# unconditional `fprintf` showed exactly two `peek=='/'` calls in the
# entire file -- both at the typed_literal `:`/path boundary on
# line 259 -- and zero calls at line 150 col 21 where the bare
# `/allow.txt` begins. Tree-sitter's external scanner is invoked
# only when an external token is in `valid_symbols`, so the
# observable conclusion is that `_absolute_path_literal` is *not*
# in valid_symbols at the LR state reached after `Path path6 = `,
# even though the rule is wired into `_literal` (which is in
# `_expression`).
#
# Tried (none worked end-to-end):
#   - `absolute_path_literal: $ => $._absolute_path_literal` wrapper
#     in `_literal` -- scanner not called at line 150.
#   - `alias($._absolute_path_literal, $.absolute_path_literal)`
#     directly in `_literal` -- same result.
#   - `prec.dynamic(2, $._absolute_path_literal)` on the wrapper to
#     force GLR resolution -- same result.
#   - Adding `absolute_path_literal` as an alternative in
#     `typed_literal`'s value choice (alongside `path_literal`):
#     introduces a regression -- `Type:/path` parses as
#     `else_expression(Type, absolute_path_literal)` instead of
#     `typed_literal(Type, path_literal)`. typed_literal's `prec(2)`
#     does not outweigh else_expression's `prec.right(0)` here, even
#     though it should -- another instance of LALR committing to a
#     non-reduction-conflict path before precedence can apply.
#   - Removing the internal `path_literal` regex entirely and
#     relying on the external scanner: same regression as above.
#
# The line-150 diagnostic suggests the missing piece is in how
# tree-sitter's grammar generator computes which states have the
# external token in valid_symbols. The expected path
# `_expression -> _literal -> absolute_path_literal -> _absolute_path_literal`
# should propagate the external into the LR state at `Path path6 = `,
# but observably does not. A future attempt should:
#   (1) inspect parser.c's lex_state tables to confirm exactly which
#       states list `sym__absolute_path_literal` in their actions, and
#   (2) compare those state IDs against the parser-state trace at the
#       line-150 position (e.g. via `tree-sitter parse --debug`) to
#       see which state the parser actually reaches there.
# It is possible that the issue is a TOKEN PRIORITY interaction with
# the internal lexer: when an internal token at the same position has
# higher implicit priority, tree-sitter may skip the external scanner.
