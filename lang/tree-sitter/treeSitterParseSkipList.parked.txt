# === C5 — nested angle-bracket function types (LHS fixed) ===
#
# Both sides now parse. Closing-side via TYPE_GT external scanner
# (commit d384f2394). Opening-side via splitting the binary shift
# operators `<<` / `>>` / `>>>` into `seq('<', token.immediate('<'))`
# (etc.) so the lexer never produces a multi-char shift token. With
# `<<` no longer a terminal, the lexer naturally emits `<` `<` for
# `Function<<Int, String>, <Int>>`, and the type_arguments /
# angle_bracket_type_list rules consume the chars normally. Cost: a
# 1-element self-conflict on `binary_expression` -- at state
# `_expression • >` the parser cannot tell from a single lookahead
# whether `>` opens a precedence-9 comparison, a precedence-10 range,
# or a precedence-11 shift, so GLR forks and unmatched alternatives
# die on the second char. Sweep stays at 872/872.
#
# Note: assignment shifts (`<<=` / `>>=` / `>>>=`) are deliberately
# NOT split. The lexer's longest-match rule still produces them as
# single 3-char tokens (since `<<` is no longer a terminal, the
# `<<=` DFA path is the only multi-char match starting with two
# `<`s). Splitting them would create an LALR(1) ambiguity at
# `_expression • <` between the assignment and binary-shift paths
# (verified on a discarded attempt -- the validator reported a 5-way
# conflict spanning assignment_expression, binary_expression,
# unary_expression, call_expression, async_call_expression).
#
# generics.x parses cleanly and is removed from this list.
#
# reflect.x stays parked: the C5 LHS errors at lines 31-32 are gone,
# but a previously-masked, unrelated bug surfaces at lines 193+. The
# parser inside a template interpolation `{f2 } -> {f2 ()}` greedily
# commits to a lambda `f2 -> {f2 ()}` whose body block consumes the
# `}` that should have closed the first interpolation. Minimal
# repro: `console.print($"x = {y } -> {y ()}");`. Fix likely needs
# either a higher precedence on the `_template_expr_end` reduction,
# or a conflict declaration that keeps the
# interpolation-end-vs-block-start interpretation alive.

# === callTests.x — bare-ident + typed-decl in tuple_assignment ===
#
# Failing shape: `(i1, Int i2) = testMultiReturns(0);` -- a tuple
# assignment whose first element is a bare existing-variable
# identifier (`i1` was declared earlier) and whose second is a fresh
# typed declaration (`Int i2`). The grammar's
# `tuple_assignment_element` alternation accepts both the typed form
# (`Type identifier`) and bare expressions (`$._expression`), so both
# elements *should* match.
#
# However, GLR consistently commits to the alternative interpretation:
# `(i1, Int i2)` parses as a `tuple_type` (with `i1` read as a
# type_name and `Int i2` as a typed tuple_type_element), making the
# whole line a `variable_declaration` with `(i1, Int i2)` as the type
# and `testMultiReturns` as the variable name -- which then ERRORs
# on `=`.
#
# Tried (none worked):
#   - `prec(2)` on `tuple_assignment` itself
#   - `prec.dynamic(1)` on the `$._expression` alt of `tuple_assignment_element`
#   - 4-way conflict adding `_for_tuple_destructure_element`
#   - `prec.dynamic(10)` on the entire `tuple_assignment_element` rule
#   - `prec.dynamic(10, prec(2, ...))` on the entire `tuple_assignment` rule
#     (parse time tripled, confirming GLR explored more paths, but
#     the result was unchanged -- prec.dynamic is not consulted at
#     the deterministic-choice point)
#   - explicit `[$.tuple_assignment, $.variable_declaration]` conflict
#     -> tree-sitter reports "unnecessary conflict"
#   - explicit `[$.tuple_assignment, $.tuple_type]` and
#     `[$.tuple_assignment, $.parenthesized_type]` conflicts
#     -> both reported as "unnecessary conflicts"
#
# Diagnostic insight from the "unnecessary conflict" warnings:
# tree-sitter's LALR(1) generator never sees a generation-time
# conflict between these container-level rules. It commits to
# `variable_declaration` (with `(...)` as a tuple_type type_expression)
# deterministically when seeing `(` at statement start. The
# element-level conflict declarations and prec.dynamic biases never
# get a chance to apply because the container-level decision was
# already made.
#
# The fix likely requires either:
#   (a) a structural rewrite that disambiguates earlier (e.g. lift the
#       `(bare_ident, Type ident, ...) = ...` shape into its own
#       container that the LALR generator can statically distinguish);
#   (b) restructuring `_expression` and `type_expression` so the bare
#       `identifier` reduction in tuple-assignment context goes
#       through a hidden non-type rule (alias/hidden);
#   (c) a workaround in `tuple_assignment_element` that uses an alias
#       with high-prec.dynamic so the bare-ident path doesn't even
#       reduce as identifier-via-type-name.
