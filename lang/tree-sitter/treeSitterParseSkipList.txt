# === (2) PARKED — nested angle-bracket function types (C5, half-fixed) ===
#
# Failing shapes: `Function<<Int, String>, <Int>>` (reflect.x line 32),
# `TestTurtle<<Int, String>>` (generics.x line 100). Both appear in
# variable_declaration position (`Type<<...>> name = expr;`).
#
# Closing-side fix shipped: external scanner token TYPE_GT splits `>>` (and
# friends) when the parser is inside a type_arguments / type_parameters /
# angle_bracket_type_list and asks for a closing `>`. This unblocks the RHS
# shapes -- e.g. `new TestTurtle<<Int, String>>()` (generics.x line 100,
# right-hand side) and `bar.as(Function<<Int, String>, <Int>>)` (reflect.x
# line 33, right-hand side) now parse cleanly. See commit d384f2394.
#
# Opening-side fix attempted but reverted: a symmetric TYPE_LT external
# token broke 160 files. The reason is asymmetric ambiguity. The CLOSING
# `>` is unambiguous at parse time -- the parser only requests TYPE_GT
# inside type-position rules, where `>>` cannot mean right-shift. The
# OPENING `<` is ambiguous at parse time -- whenever the parser has both
# a variable_declaration alternative and an expression_statement
# alternative alive (which is essentially every statement starting with
# an identifier), TYPE_LT becomes a "valid" symbol because the
# variable_declaration path could enter type_arguments. The scanner then
# eagerly consumes `<`, killing the comparison interpretation. Result:
# `a < b` everywhere broke.
#
# A smarter TYPE_LT that only fires when the parser is UNAMBIGUOUSLY in
# type-arguments context (e.g. after `new`, after `.as(`, in a parameter
# type position) might work, but tree-sitter's external scanner only
# sees `valid_symbols`, not parser-state context. The cleanest path is
# probably a structural grammar rewrite that disambiguates the
# variable_declaration / expression_statement choice earlier (similar
# shape to the callTests.x puzzle below).
#
# See lsp-bugfixes-gene-plan.md "C5 — reflect.x ... (parked)".

manualTests/src/main/x/reflect.x
manualTests/src/main/x/generics.x

# === (2c) PARKED — callTests.x: bare-ident + typed-decl in tuple_assignment ===
#
# Failing shape: `(i1, Int i2) = testMultiReturns(0);` -- a tuple
# assignment whose first element is a bare existing-variable identifier
# (`i1` was declared earlier) and whose second is a fresh typed
# declaration (`Int i2`). The grammar's `tuple_assignment_element`
# alternation accepts both the typed form (`Type identifier`) and bare
# expressions (`$._expression`), so both elements *should* match.
#
# However, GLR consistently commits to the alternative interpretation:
# `(i1, Int i2)` parses as a `tuple_type` (with `i1` read as a type_name
# and `Int i2` as a typed tuple_type_element), making the whole line a
# `variable_declaration` with `(i1, Int i2)` as the type and
# `testMultiReturns` as the variable name -- which then ERRORs on `=`.
#
# Tried (none worked):
#   - `prec(2)` on `tuple_assignment` itself
#   - `prec.dynamic(1)` on the `$._expression` alt of `tuple_assignment_element`
#   - 4-way conflict adding `_for_tuple_destructure_element`
#   - `prec.dynamic(10)` on the entire `tuple_assignment_element` rule
#   - `prec.dynamic(10, prec(2, ...))` on the entire `tuple_assignment` rule
#     (parse time tripled, confirming GLR explored more paths, but
#     the result was unchanged -- prec.dynamic is not consulted at
#     the deterministic-choice point)
#   - explicit `[$.tuple_assignment, $.variable_declaration]` conflict
#     -> tree-sitter reports "unnecessary conflict"
#   - explicit `[$.tuple_assignment, $.tuple_type]` and
#     `[$.tuple_assignment, $.parenthesized_type]` conflicts
#     -> both reported as "unnecessary conflicts"
#
# Diagnostic insight from the "unnecessary conflict" warnings:
# tree-sitter's LALR(1) generator never sees a generation-time
# conflict between these container-level rules. It commits to
# `variable_declaration` (with `(...)` as a tuple_type type_expression)
# deterministically when seeing `(` at statement start. The
# element-level conflict declarations and prec.dynamic biases never
# get a chance to apply because the container-level decision was
# already made.
#
# The fix likely requires either:
#   (a) a structural rewrite that disambiguates earlier (e.g. lift the
#       `(bare_ident, Type ident, ...) = ...` shape into its own
#       container that the LALR generator can statically distinguish);
#   (b) restructuring `_expression` and `type_expression` so the bare
#       `identifier` reduction in tuple-assignment context goes through
#       a hidden non-type rule (alias/hidden);
#   (c) a workaround in `tuple_assignment_element` that uses an alias
#       with high-prec.dynamic so the bare-ident path doesn't even
#       reduce as identifier-via-type-name.
#
# Parked because it's a single-file edge case and the rest of the
# destructuring fix unblocks 15 of 16 files. Revisit when the next
# parser-state debugging session has cycles.

manualTests/src/main/x/jit/jit_tests/basic/callTests.x
