=========================

This file documents changes made during the AST refactoring effort on the
lagergren/reflect branch.

Session 1: Copyable and PoolTransferable Implementation
================================================================================

1. COPYABLE INTERFACE FOR AST NODES
-----------------------------------
File: javatools/src/main/java/org/xvm/compiler/ast/AstNode.java

- Made AstNode implement Copyable<AstNode>
- Added copy() method that returns AstNode (not generic <T extends AstNode>)
- copy() creates deep copy of AST subtree using forEachChild() and withChildren()

Problem encountered: Signature mismatch between Copyable<AstNode>.copy() returning
AstNode and the desired <T extends AstNode> T copy() for type-safe copies.

Solution: Use covariant return types in intermediate classes.


2. COVARIANT COPY() OVERRIDES
-----------------------------
Added @Override copy() methods returning the specific type to these classes:

- Expression.java: public Expression copy() { return (Expression) super.copy(); }
- Statement.java: public Statement copy() { return (Statement) super.copy(); }
- StatementBlock.java: public StatementBlock copy() { ... }
- TypeExpression.java: public TypeExpression copy() { ... }
- TypeCompositionStatement.java: public TypeCompositionStatement copy() { ... }
- PropertyDeclarationStatement.java: public PropertyDeclarationStatement copy() { ... }
- UnaryComplementExpression.java: public UnaryComplementExpression copy() { ... }

This allows type-safe copies without casts at most call sites:
  Expression exprCopy = someExpr.copy();  // No cast needed


3. POOLTRANSFERABLE INTERFACE FOR CONSTANTS
-------------------------------------------
File: javatools/src/main/java/org/xvm/asm/Constant.java

- Made Constant implement PoolTransferable<Constant>
- Removed Cloneable interface (we're replacing clone() with explicit copy/transfer)
- Added copy() that returns this (constants are immutable, no copy needed)
- Added transferTo(ConstantPool) that throws UnsupportedOperationException
  (requires subclass override)
- Changed adoptedBy() to throw UnsupportedOperationException
  (deprecated in favor of transferTo())


4. TRANSFERTO() IMPLEMENTATIONS
-------------------------------

5. CONSTANTPOOL.REGISTER() UPDATED
----------------------------------
File: javatools/src/main/java/org/xvm/asm/ConstantPool.java

Changed register() to use transferTo() instead of adoptedBy():

if (constant.getContaining() != this) {
    // Transfer the constant to this pool - transferTo() handles registration
    return (T) constant.transferTo(this);
}

// And for locators:
if (oLocator instanceof Constant constLocator && constLocator.getContaining() != this) {
    constLocator = constLocator.transferTo(this);
    oLocator = constLocator;
}


6. POOLTRANSFERABLE.TRANSFERARRAY() FIXED
-----------------------------------------
File: javatools/src/main/java/org/xvm/asm/PoolTransferable.java

Fixed generic bounds conflict. Changed from:
  <C extends Constant & PoolTransferable<C>> C[] transferArray(...)

To:
  <C extends Constant> C[] transferArray(...)

With internal cast since Constant now implements PoolTransferable<Constant>.

11. RECURSIVE INITIALIZATION FIX
--------------------------------
Problem: ConcurrentHashMap.computeIfAbsent() doesn't allow recursive calls.
During constant registration, creating a StringConstant would trigger type
resolution which eventually called back to ensureStringConstant().

Stack trace:
  ensureStringConstant("foo")
  -> computeIfAbsent -> register() -> registerConstants()
  -> getType() -> typeString() -> ... -> ensureStringConstant("ecstasy.xtclang.org")
  -> computeIfAbsent -> DEADLOCK (recursive update)

Root cause: ValueConstant.registerConstants() was eagerly registering the type:
  @Override
  protected void registerConstants(ConstantPool pool) {
      pool.register(getType());  // Triggers recursive constant creation
  }

File: javatools/src/main/java/org/xvm/asm/constants/ValueConstant.java (lines 43-46)

Solution: Removed the registerConstants() override entirely. For value constants
(StringConstant, IntConstant, etc.), the type is implicit from the format and
doesn't need eager registration. It's computed lazily when getType() is called.

Before:
  @Override
  protected void registerConstants(ConstantPool pool) {
      pool.register(getType());
  }

After:
  // Note: Subclasses that have explicit type fields (like ArrayConstant, MapConstant)
  // must override registerConstants() to register their type. For most value constants,
  // the type is implicit from the format and doesn't need registration here.
  // Registering the type eagerly causes circular dependencies during pool initialization.

Note: ArrayConstant, MapConstant, MatchAnyConstant already override registerConstants()
to register their explicit m_constType field, so they continue to work correctly.


12. PLAN DOCUMENT: LAZY EVALUATION ARCHITECTURE
-----------------------------------------------
File: docs/ast-refactoring-plan.md (lines 4885-5160)

Added "Lazy Evaluation and Immutability Architecture" section covering:

Core Principles:
- Immutable by default, copy-on-write transforms
- Lazy evaluation with memoization
- External caches vs embedded state

Avoiding Recursive Initialization:
- Separate creation from registration
- Lazy type resolution
- Safe patterns for ConcurrentHashMap

Stateless Stage Processing:
- Pure function transformations
- Benefits for LSP integration

Cache Hierarchy for LSP:
- Parse cache (per file)
- Type cache (global)
- Semantic cache
- Invalidation strategies

Migration Path (5 phases):
1. Fix immediate issues (done: ValueConstant, typed maps)
2. Introduce Lazy patterns (Lazy<T> utility)
3. Copy-on-Write AST (withX() methods)
4. External caches (TypeInfoCache extraction)
5. Stateless stages (visitor refactoring)


13. PLAN DOCUMENT: CONSTANTPOOL CLEANUP
---------------------------------------
File: docs/ast-refactoring-plan.md (lines 5164-5380)

Added "ConstantPool Cleanup: From 4000 Lines to Clarity" section covering:

Analysis:
- 191 ensure* methods with repeated patterns
- ~4000 lines of similar code

Proposed Refactoring:
1. Typed ensure pattern - generic helper method
2. Null handling at boundaries - Objects.requireNonNull
3. Separate concerns with private helpers
4. Builder pattern for complex type construction
5. Group related methods logically
6. Consider extracting factories

Migration Strategy:
- Don't refactor all 191 methods at once
- Start with most-used methods
- Add typed maps incrementally
- Extract helpers first, keep API stable
- Add tests before each refactoring

Code Size Goals:
| Current     | Goal        | Reduction |
|-------------|-------------|-----------|
| ~4000 lines | ~1500 lines | 60%       |
| 0 typed maps| ~20 maps    | Type safe |
| ~50 null checks | ~10     | Clarity   |

14. PLAN DOCUMENT: LEXER, PARSER, COMPILER CLEANUP
--------------------------------------------------
File: docs/ast-refactoring-plan.md (lines 5383-5713)

Added comprehensive "Lexer, Parser, and Compiler Cleanup" section covering:

Overview Statistics:
| File           | Lines | Methods | Avg Lines/Method | Null Checks |
|----------------|-------|---------|------------------|-------------|
| Lexer.java     | 2,859 | ~50     | ~57              | 18          |
| Parser.java    | 5,772 | 40      | ~144             | 268         |
| Compiler.java  | 1,373 | ~30     | ~46              | ~20         |

Lexer Analysis:
- Giant switch in eatToken() (~620 lines)
- Proposed dispatch table + handler methods pattern
- LSP considerations (incremental lexing, error recovery)

Parser Analysis:
- God method: parseClassExpression() (~1,650 lines!)
- 268 null checks scattered throughout
- Mutable state prevents caching/parallelism
- Proposed Pratt parser with precedence table
- LSP-friendly IncrementalParser interface

Compiler Analysis:
- Current state machine architecture (12 stages)
- Issues: stateful transitions, single-module coupling
- Proposed stateless CompilationPhase<I,O> pipeline
- LSPCompilationCache for incremental compilation

Cross-Cutting Concerns:
1. God Classes Problem - Split by Single Responsibility
2. Input Sanitization Layer - Validation at API boundaries
3. Immutability for LSP - withX() methods, persistent maps
4. Performance Opportunities:
   - Parallel module compilation (2-4x gain)
   - Incremental parsing (10-100x for edits)
   - External TypeInfo caching
   - Typed constant maps

Migration Priority:
- High: Parser error recovery, stateless phases, immutable AST
- Medium: Split god classes, extract handlers, validation layer
- Lower: Parallel compilation, incremental parsing, cache optimization
